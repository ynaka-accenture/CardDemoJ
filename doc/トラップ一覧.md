# 日本版 CardDemo (J-CardDemo) — 設計とトラップ一覧

## 目的

AWS CardDemo を日本のメインフレーム業務に寄せて改造し、**簡単にはUTF-8へ
移行できない「日本語固有のトラップ」**を意図的に埋め込んだモダナイゼーション
テストベッドを構築する。シングルバイトのみの本家CardDemoでは見えない、
IBMメインフレーム(z/OS)の EBCDIC日本語(IBM漢字)資産の移行困難性を再現する
(同様のSO/SI方式は富士通JEF・日立KEIS・NEC JIPSでも用いられる)。

## 文字コードモデル

J-CardDemo は **IBM EBCDIC日本語(IBM漢字 / DBCS-Host, CCSID 930/939/5035)** を原資産の文字コードとする。
DBCS(2バイト文字=漢字・かな)は SO(Shift-Out 0x0E)/SI(Shift-In 0x0F)で
囲まれた「混在フィールド」として PIC X 領域に格納される。

```
混在フィールド X(20) の例:
  バイト列: [SO][漢][字][SI] A B C ...
            0E  XXXX XXXX 0F  41 42 43
  → SO/SI が各1バイトを消費し、漢字は各2バイト
```

## 埋め込んだトラップ一覧(全11種)

### トラップ1: 固定バイト幅 vs 多バイト文字
COBOL `X(n)` は **n バイト固定**。EBCDIC/SJISで漢字=2バイト、UTF-8では3バイト。
`X(20)` に収まっていた「漢字10文字(20バイト)」は UTF-8 では30バイトとなり
**フィールド長を超過**する。フィールド定義の再設計なしに移行不可。

### トラップ2: SO/SI シフトコード
JEF/KEIS の混在フィールドは SO(0x0E)/SI(0x0F)で DBCS 区間を囲む。
- 単純なバイト→UTF-8変換ではシフトコードがゴミ文字(制御文字)として残る
- シフトコードはフィールド内のバイト数を消費するため、有効文字数の計算が
  「(全体長 - SO/SIの数*1 - 単バイト数) / 2」という非自明な式になる
- ネストした SO...SO...SI のような壊れた状態(後述トラップ9)も実在する

### トラップ3: PIC N(各国語項目)と USAGE NATIONAL
`PIC N(10)` は DBCS 10文字。COBOL内部表現は処理系依存:
- DISPLAY扱い: SJIS/EBCDIC で 20バイト
- NATIONAL扱い: UTF-16 で 20バイト(ただしホスト依存)
N項目とX項目混在フィールドが同一レコードに併存し、移行時の型判定を困難にする。

### トラップ4: ベンダー外字(Gaiji)・機種依存文字
JEF 拡張漢字・利用者定義文字(外字)は Unicode に正準マッピングが無く、
私用領域(PUA U+E000〜)へ退避するしかない。本テストベッドでは外字コード
ポイントを顧客名に埋め込み、ラウンドトリップ不能ケースを再現する。

### トラップ5: 波ダッシュ/全角チルダ問題
SJIS 0x8160 は Unicode で U+301C(波ダッシュ)か U+FF5E(全角チルダ)か
変換系で揺れる。住所の「〜丁目」表記で発生させる。

### トラップ6: 円記号/バックスラッシュ(0x5C)問題
EBCDIC/JIS X 0201 で 0x5C は円記号(¥)。ASCII/UTF-8 ではバックスラッシュ(\)。
金額表記「¥」を単バイト 0x5C で格納し、移行時の文字化けを誘発する。

### トラップ7: 半角カナ(JIS X 0201 片仮名)
1バイト片仮名(0xA1〜0xDF)。SO/SI 不要だが ASCII 範囲外の単バイト。
UTF-8 では3バイトの全角カナへ化けるか、マッピング表が必要。電話番号メモ欄に使用。

### トラップ8: DBCS空白パディング(0x4040 / 0x8140)
混在フィールドの余白を DBCS空白(EBCDIC 0x4040 / SJIS 0x8140)で埋める。
単バイト空白(0x20/0x40)と混在し、TRIM 処理やパディング判定を狂わせる。

### トラップ9: SO/SI 不整合(壊れたデータ)
実務データには SO はあるが SI が無い、奇数バイトで DBCS が途切れる等の
不整合が紛れ込む。素朴な変換はここで例外を出すか黙って破損する。

### トラップ10: 全角数字・全角英字
金額や ID に全角数字(０-９ = SJIS 0x824F〜)が混入。数値編集・比較で
半角数字と区別が必要。NUMVAL 等が誤動作する。

### トラップ11: コラーティングシーケンス(整列順)依存
EBCDIC 整列順は ASCII/Unicode と異なる(数字と英字の順序が逆等)。
SORT/比較ロジックが文字コードの整列順に暗黙依存しており、移行で順序が変わる。

## 移行困難性の実証方針

各トラップについて、「素朴なUTF-8一括変換」が失敗することを示す:
1. バイト長超過(トラップ1) → レコード長オーバーフロー
2. SO/SI残存(トラップ2,9) → 制御文字混入・文字数誤算
3. 外字/波ダッシュ/円記号(トラップ4,5,6) → ラウンドトリップ不能・文字化け
4. 整列順(トラップ11) → SORT結果の順序変化

これらは GnuCOBOL 対比の動的テストでは「単純比較」では�捉えられないため、
文字コード・バイトレイアウト・整列順を含む専用の検証が必要であることを示す。
