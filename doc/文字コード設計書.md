# 文字コード設計書 — 日本版CardDemo

## 1. 文字コードモデル

日本版CardDemo は IBM メインフレーム(z/OS + CICS / IMS / DB2 / MQ)を
ターゲットとするため、**IBM EBCDIC日本語(IBM漢字 / DBCS-Host)** を原資産の
文字コードとする。CICS BMS の DBCS フィールド、IMS セグメント、DB2 の DBCS 列、
MQ の CCSID 変換は、いずれもこの IBM の文字コード体系を前提とする。

| 文字コード体系 | 用途 | 備考 |
|--------------|------|------|
| IBM EBCDIC日本語(IBM漢字) | 本テストベッドの基準 | DBCS-Host。SO/SI混在。CCSID 930(カナ)/939(英小文字)/5026/5035 |
| CCSID 1208 | 移行先 | UTF-8 |
| CCSID 943 | 参考 | Shift-JIS(分散環境との連携時) |

参考: 国産メインフレームでも同様の SO/SI 混在方式が用いられる
(富士通 JEF、日立 KEIS、NEC JIPS)。移行困難性の本質は IBM・国産で共通だが、
本テストベッドは IBM メインフレームを対象とするため IBM 漢字を基準とする。

## 2. 混在フィールドの構造

漢字と英数字が混在するフィールドは、DBCS区間を SO(Shift-Out)/SI(Shift-In)
で囲む。

```
バイト列:  [SO] 漢  字 [SI]  A  B  C
16進:      0E  XXXX XXXX 0F  41 42 43
意味:      区間  2バイト文字  区間  単バイト文字
           開始              終了
```

- SO(0x0E): 以降は2バイト文字(DBCS)
- SI(0x0F): 以降は1バイト文字(SBCS)
- 漢字1文字 = 2バイト(EBCDIC) / UTF-8では3バイト

## 3. 有効文字数の計算

混在フィールドの文字数は `LENGTH` では求まらない。SO/SIを解釈し、
DBCS区間は2バイトで1文字、SBCS区間は1バイトで1文字として数える。

```
有効文字数 = SBCS文字数 + DBCS文字数
DBCS文字数 = (DBCS区間の総バイト数) / 2
```

この計算は共通サブルーチン `漢字解析`(英字名: JCKANJ0C)が担う。

## 4. UTF-8移行時の課題(層別)

| 層 | 内容 | 移行可否 |
|----|------|---------|
| 層1 | ソースファイルのエンコーディング | UTF-8化容易 |
| 層2 | I/O境界のCharset明示 | UTF-8明示可能 |
| 層3 | レコード内データのバイト表現 | **構造的制約あり** |

層3の壁:
- `X(n)` は n バイト固定。漢字はEBCDIC2バイト→UTF-8 3バイトで枠超過
- SO/SIシフトコードは UTF-8 に存在しない概念(除去が必要)
- 外字は Unicode 正準マッピングが無く、私用領域(PUA)へ退避
- フィールド幅の再設計(`X(n)`→可変長 or 幅拡張)が不可避

## 5. ミドルウェア境界の文字コード

| ミドルウェア | 境界 | 論点 |
|-------------|------|------|
| MQ | MQMD の CodedCharSetId | 930(EBCDIC) vs 1208(UTF-8)。MQGMO-CONVERTの変換テーブル |
| DB2 | 列の CCSID | EBCDIC混在列とUTF-8 DB2の不一致。VARCHAR桁あふれ |
| IMS | DBDのセグメント | バイト列として透過。文字コード変換はアプリ責務 |

## 6. 推奨移行アプローチ

1. **Phase 1(Lift)**: バイトレイアウトを保ったまま確実に移行。SO/SIも温存
2. **意図への遡行**: 各フィールドの文字コード意味(DBCS/SBCS/外字)を確定
3. **Phase 2(Shift)**: フィールド幅を再設計しUTF-8/Unicodeネイティブへ再構築
4. **等価性ゲート**: 文字コード・バイトレイアウト・整列順を含む専用テストで検証
